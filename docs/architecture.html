YOLO11 · RTSP · Email

工業煙囪冒煙監測系統：專案架構

給第一次接觸 AI 視覺辨識者的設計說明。版本 0.1.0，適用於先在訓練電腦微調模型，再部署到只能存取內部攝影機與郵件 relay 的現場電腦。

先講結論：這不是「看到一張圖有白色就寄信」的系統。它先用 YOLO11-seg 找出煙霧的形狀，再用連續影格確認煙霧持續存在，最後才寄出 Email。這三層分開，才能逐步調整速度、準確率與誤報率。

第一次接觸 YOLO11 的讀者，可先閱讀 YOLO11 特色與工作原理導覽，再回來看本文件如何把模型接到 ROI、RTSP、時間狀態機與 Email。

1. 系統要解決的問題

輸入是攝影機的 RTSP 串流，例如 rtsp://user:password@10.34.23.152:554/live.sdp。每隔固定時間抽一張影格，判斷指定煙囪附近是否有煙霧；若連續多張影格都符合條件，便儲存證據影像並經由 SMTP 寄送通知。

辨識目標

一個類別：smoke。IJmond 原始遮罩的低／高不透明度在第一版合併，因為目前需求是「是否冒煙」，不是判定 EPA Method 9 等級。

資料來源

IJmond Industrial Smoke Segmentation Dataset。它包含像素遮罩與 polygon 資訊；本專案把 raster mask 轉成 YOLO polygon label。

部署邊界

模型、程式與紀錄留在現場電腦。RTSP 與 SMTP relay 必須是可達的內部網路資源；隔離外網不代表可以直接寄到公開信箱。

3. 為什麼使用 YOLO11-seg

YOLO11 是 Ultralytics 的即時視覺模型系列，支援 detection、instance segmentation 等任務；分割模型的檔名使用 -seg，例如 yolo11n-seg.pt。本專案使用預訓練權重再用工業煙霧資料微調，而不是從隨機權重開始訓練，通常可以更快學到輪廓與背景特徵。

模型特性建議用途
yolo11n-seg.pt最小、速度快、顯存需求較低第一次建立基線；8 GB GPU 或現場 CPU 優先
yolo11s-seg.pt較大的特徵抽取能力，速度與顯存需求中等n 版漏檢較多時，以本地煙囪資料微調
yolo11m/l/x-seg.pt更大、訓練與推論成本高只有在驗證資料證明值得時才考慮

「模型大小越大一定越好」是不正確的。對這個專案，現場攝影機角度、光線、煙囪外觀與 ROI 是否相似，通常比把模型從 n 換成 l 更重要。

4. 資料集與資料轉換

IJmond 資料集來自三個固定攝影機，提供 cropped images、mask、metadata 以及依時間或攝影機切分的清單。資料集頁面列出 900 張 raw images、1209 個 polygon，以及 2074 張 cropped images，其中 1109 張有煙霧。原始 mask 以 0 表示背景、155 表示低不透明度煙霧、255 表示高不透明度煙霧。

本專案的 scripts/prepare_ijmond.py 會：

  1. 優先讀取資料集提供的 split_by_camera 或 split_by_timestamp 清單；
  2. 將 mask 中所有大於 0 的像素合成二值 smoke mask；
  3. 以 OpenCV 找出連通區域輪廓，正規化成 YOLO segmentation label；
  4. 保留無煙影像，產生空的 .txt label，讓模型學會「沒有煙」；
  5. 輸出 images/{train,val,test} 與 labels/{train,val,test}。

預設使用 camera split 的目的，是讓測試較接近「換一個攝影機／背景後仍能否辨識」。但使用者的實際工廠與荷蘭資料集不是同一場域，所以 IJmond 測試分數不能直接當作現場保證；必須另外用現場影片驗收。

5. 現場推論與告警判斷

每次模型預測會得到煙霧 mask、偵測信心度與煙霧面積比例。程式先判定單張影格是否為 positive：

positive = (存在 smoke 實例)
        且 (信心度 ≥ CONF_THRESHOLD)
        且 (煙霧遮罩面積 / ROI 面積 ≥ MIN_SMOKE_AREA_RATIO)

接著由狀態機處理時間：

狀態條件動作
NORMAL連續 positive 達到 POSITIVE_FRAMES_TO_ALARM進入 ALARM、儲存證據、寄送開始警報
ALARM煙霧仍持續，且距離上次通知超過 ALERT_REPEAT_SECONDS寄送持續警報，避免每秒寄信
ALARM連續 negative 達到 NEGATIVE_FRAMES_TO_CLEAR回到 NORMAL，可寄送解除通知

例如抽樣間隔是一秒、positive 門檻是 5，代表至少約 5 秒持續判定為煙霧才告警。這不是物理安全聯鎖，也不是排放法規判定；它是影像事件通知系統。

7. 失效模式與安全邊界

RTSP 斷線

讀取器釋放連線並重連；不把斷線當成「無煙」，因此不會因鏡頭失效而寄送解除通知。

SMTP 故障

監測程序繼續執行，失敗資訊寫入 detections.jsonl 與 log。部署前必須先測試內部 relay，並建立 log 監看。

模型誤判

調整 ROI、信心度、面積比例與連續影格門檻；必要時加入現場標註資料微調，不要只盲目增加 epochs。

法規與人身安全

影像告警不應取代現場安全聯鎖、人工確認或法定排放量測。Email 延遲與網路故障都可能發生。

8. 監控輸出與人工複核

設定 ALLOW_LIVE_STREAMING=true 時，程序會在記憶體中啟動 HTTP frame stream，不寫入 live JPG。瀏覽器開啟 http://127.0.0.1:8765/ 後，現代瀏覽器會以 HTML5 <video> 元件播放即時標註影片，內容包含 ROI、box、mask、conf、area、cls；分數也可由 /latest.json 讀取。底層 frame transport 使用標準 multipart JPEG，瀏覽器再以 canvas.captureStream() 產生 MediaStream，因此不依賴瀏覽器直接播放 MJPEG。IE／IE 相容模式不支援這組現代 API，頁面會自動改用 ES5 XMLHttpRequest + canvas 相容預覽；現場若可行，建議使用最新版 Edge 的非 IE 模式。預設只綁定 localhost；若要讓同網段其他電腦觀看，才把 LIVE_STREAM_HOST 改為 0.0.0.0，並由 IT 設定防火牆。

偵測為 smoke 時，預設會把標註影像與詳盡 JSON sidecar 保存到 data/runtime/alerts/YYYY-MM-DD/，並在 JSONL 寫入完成後以 INFO 推送 conf、area、instance、快照狀態與檔案路徑到 CLI 及 data/runtime/logs/monitor.log。主記錄 detections.jsonl 預設只保存煙霧偵測與告警／解除事件，不會逐張保存無煙影格；如需無煙稽核，可用 NO_SMOKE_LOG_INTERVAL_SECONDS 開啟週期摘要。資料夾與訓練資料集分開，使用 報警資料檢查／重標註子專案 可把人工確認的樣本保留、移動到 YOLO 資料集或刪除。SHADOW_MODE=true 仍會執行推論、狀態機與保存煙霧／事件，但會抑制所有 Email，適合新模型正式切換前驗證。

9. 現場部署與維運

現場 Windows 電腦採用 Python 3.11.15 與已驗證的相依套件執行原始程式。隔離環境先依離線安裝手順準備 Windows wheel／Conda 套件，再複製程式、models/best.pt 與受控建立的 configs/monitor_settings.env。本 repository 不包含 Windows 可執行檔打包流程。

部署驗收應依序確認 Python／套件可匯入、模型可載入、RTSP 能穩定取流、SMTP relay 可達，以及監測程序重啟後能恢復。若公司端點政策禁止未核准腳本或套件，應先由 IT 核准 wheelhouse、版本清單與檔案雜湊。
