工業煙囪冒煙監測系統：從訓練到部署操作書

適用於 YOLO11-seg、IJmond Industrial Smoke Segmentation Dataset、RTSP 攝影機與 Email 告警。指令預設在專案根目錄執行；Windows PowerShell 與 Linux shell 都可使用，路徑分隔符號依作業系統調整。

最重要的部署觀念：將工作分成兩台電腦。訓練電腦可以連外，用來取得資料集、Python 套件與初始 YOLO11 權重；現場電腦可以不連外，只需存取內部 RTSP 攝影機與內部 SMTP relay。把模型與已驗證的 Python 套件以受控媒介帶進現場。

Windows 離線 CPU 安裝：若目標是 Windows AMD64、Python 3.11.15 的隔離電腦，請先閱讀 Python 3.11.15 與 YOLO11 CPU 離線安裝手順。主專案 requirements.txt 同時保留 GPU 與 CPU 選項；CPU 安裝前必須將 CUDA torch 行註解，並取消 CPU torch 行的註解。

模型入門：如果你還不熟悉 YOLO11 的 Detect、Segment、訓練、mAP 與部署格式，請先閱讀 YOLO11 特色與工作原理導覽；本操作書再說明這些概念如何套用到煙囪冒煙監測。

0. 你需要先知道的名詞

名詞白話解釋
影格 frame影片中的一張圖片。程式不是一次理解整段影片，而是逐張或每隔幾張取圖。
ROIRegion of Interest，感興趣區域。只裁切煙囪附近，降低天空、雲、樹影與其他設備造成的誤判。
YOLO11-seg可即時推論的 YOLO11 實例分割模型；除了矩形框，也能輸出煙霧的像素遮罩。
instance segmentation模型不只畫矩形框，也畫出每一個煙霧區域的像素輪廓。
confidence模型對「這個區域是 smoke」的信心度，不是物理上煙濃度的百分比。
mAP資料集上的綜合評分；不能直接等同於你的工廠現場準確率。
SMTP relay負責替程式轉送 Email 的郵件伺服器。隔離外網的電腦仍需能連到它。

將本資料夾命名為 industrial-smoke-monitor，建立 Git repository。原始資料集、模型權重、密碼與現場紀錄已由 .gitignore 排除。

cd industrial-smoke-monitor
git init
git add .
git commit -m "Initial industrial smoke monitoring project"
# 建立 GitHub repository 後再設定 remote 與 push

本專案以 Python 3.11.15 為主要驗證版本。先確認 NVIDIA 驅動與 GPU：

nvidia-smi

.venv\Scripts\Activate.ps1
# Linux
source .venv/bin/activate

目前其餘核心版本也要保持一致：numpy==1.26，以及標準 opencv-python>=4.9.0,!=4.13.0.90。不要在同一環境混裝 opencv-python 與 opencv-python-headless。

python -m pip install --upgrade pip

若輸出 CUDA available: False，可能是 CPU 版或 GPU 驅動／CUDA 不相容。隔離 Windows CPU 電腦請依上方離線安裝手順建立環境，不要直接把 Linux 的環境或未針對 Windows 解析的 wheelhouse 搬過去。

3. 下載並檢查 IJmond 資料集

從 Figshare 的 IJmond dataset 頁面下載 zip，放到 data/raw/ 後解壓。建議最後的結構類似：

data/raw/ijmond/
└── ijmond_seg/
    └── test/
        └── cropped/
            ├── images/
            ├── masks/
            └── splits/
                ├── split_by_camera/
                └── split_by_timestamp/

資料集官方說明中，cropped image 與 mask 使用相同的檔案 stem，mask 是 8-bit：0 為背景、155 為低不透明度煙霧、255 為高不透明度煙霧。這個專案第一版將 155 與 255 合併成同一個 smoke 類別。

4. 將資料集轉成 YOLO segmentation 格式

預設用官方 camera split。這比隨機切分更能檢查模型是否只記住某些背景；若你的研究需要時間外推，可改用 timestamp。

python scripts/prepare_ijmond.py \
  --input data/raw/ijmond \
  --output data/processed/ijmond_yolo \
  --split-strategy camera \
  --min-component-area 20 \
  --overwrite

成功後應看到：

data/processed/ijmond_yolo/
├── images/train, images/val, images/test
├── labels/train, labels/val, labels/test
├── manifest.csv
└── summary.json

查看統計：

python -c "import json; print(json.dumps(json.load(open('data/processed/ijmond_yolo/summary.json')), indent=2))"

如果 summary 顯示 0 張影像，通常是 zip 多包了一層目錄、輸入路徑錯誤，或資料夾不是官方 archive。先確認存在 cropped/images、cropped/masks 與 split txt。

5. 先以 YOLO11n-seg 建立基線

執行下列指令。第一次使用 yolo11n-seg.pt 時，Ultralytics 會在可連外的訓練電腦取得預訓練權重；如果是離線訓練，請先把該 pt 檔放入專案根目錄並將 --model 指向它。

python scripts/train.py \
  --data configs/ijmond.yaml \
  --model yolo11n-seg.pt \
  --epochs 100 \
  --imgsz 640 \
  --batch 8 \
  --device 0 \
  --workers 4 \
  --patience 25 \
  --name smoke_yolo11n

若 GPU 顯存不足，依序嘗試：減小 --batch、改 --imgsz 512、使用 yolo11n-seg 而不是 s 版、關閉 cache。不要先用更大的模型解決顯存問題。

訓練結果主要在 runs/train/smoke_yolo11n/，最佳模型通常是：

runs/train/smoke_yolo11n/weights/best.pt

train.py 會將相對的 --project 路徑固定解析到專案根目錄；即使你從 runs/segment 等其他資料夾啟動，輸出也不會再被錯誤放到 runs/segment/runs/train/。

6. 驗證模型，不要只看一個數字

先在官方 test split 驗證：

python scripts/validate.py \
  --weights runs/train/smoke_yolo11n/weights/best.pt \
  --data configs/ijmond.yaml \
  --split test \
  --imgsz 640 \
  --batch 8 \
  --device 0

檢查 box precision、box recall、box mAP 與 mask mAP。對告警系統來說，漏掉煙霧可能比多出少量候選更嚴重，但若 precision 太低，Email 會造成告警疲勞。因此需要用現場驗收影片重新權衡。

想知道分數代表什麼、下一步該改哪裡？請接著閱讀訓練結果評估與調參指南。它以本專案附上的 IJmond 100 epochs 結果為案例，示範如何看 Box／Mask 的 P、R、mAP50、mAP50-95、F1、預測圖，以及如何把模型分數轉成現場事件召回率與誤報／小時。

可用結果圖檢查遮罩是否真的蓋在煙霧上，而不是天空、建築或雲：

yolo segment predict \
  model=runs/train/smoke_yolo11n/weights/best.pt \
  source=data/processed/ijmond_yolo/images/test \
  conf=0.35 save=True

7. 用現場影片做部署前驗收

IJmond 與你的工廠在攝影機位置、煙囪外觀、季節、光線與背景上都可能不同。請先收集至少三類影片：明確冒煙、正常無煙、容易混淆（雲、蒸汽、雨、逆光、夜間）。ROI 座標可先用影像檢視工具取得。

python scripts/replay_video.py \
  --weights runs/train/smoke_yolo11n/weights/best.pt \
  --source samples/chimney_day.mp4 \
  --output data/runtime/replay_day.mp4 \
  --roi 400,80,1500,900 \
  --conf 0.35 \
  --area 0.005 \
  --sample-seconds 1 \
  --positive-frames 5 \
  --negative-frames 30 \
  --device 0

開啟輸出的 MP4，確認紅色遮罩是否覆蓋煙霧。記錄每段影片的：實際冒煙時間、模型開始告警時間、誤報次數、漏報次數、解除延遲。只有當這些數據符合現場可接受範圍，才進入部署。

9. 設定現場 configs/monitor_settings.env

複製範例並逐項修改：

copy configs/monitor_settings.env.example configs/monitor_settings.env       # Windows
cp configs/monitor_settings.env.example configs/monitor_settings.env         # Linux
欄位說明起始建議
RTSP_URL攝影機 RTSP 位址；密碼不要提交 Git使用 IT／攝影機管理員提供的內部位址
MODEL_PATH最佳權重models/best.pt
ROI_XYXYx1,y1,x2,y2，座標以原始影像左上角為原點包住煙囪出口與可能煙霧區域，留一些邊界

ALLOW_LIVE_STREAMING是否啟動記憶體內的 HTML5 video preview，不寫 live JPG需要瀏覽器即時影片時設 true
LIVE_STREAM_HOST、LIVE_STREAM_PORT串流綁定位址與連接埠127.0.0.1、8765；跨電腦觀看才考慮 0.0.0.0
SAVE_ALERT_SNAPSHOTS煙霧偵測時是否保存告警 JPG 與詳盡 JSONtrue；輸出在 data/runtime/alerts
ALERT_SNAPSHOT_INTERVAL_SECONDS連續 smoke 時保存兩筆告警的最短間隔1.0；依磁碟容量調整

SMTP_SECURITYnone、starttls 或 ssl依 IT 的 SMTP relay 政策
ALERT_TO逗號分隔收件人至少一個環保／製程責任群組
密碼安全：configs/monitor_settings.env 內的 RTSP 密碼與 SMTP 密碼視同機密。設定檔權限只給服務帳號，備份時加密；不要貼到 GitHub、聊天紀錄、截圖或 log。

10. 部署前測試 RTSP 與 Email

先測 RTSP，這一步不需要載入模型：

python scripts/check_rtsp.py --seconds 15
# 預期：收到 frames，並寫出 data/runtime/rtsp_test.jpg

再測 SMTP：

python scripts/test_email.py
# 預期：Test email sent.

若 RTSP 通而 Email 不通，這不是 YOLO 問題；檢查 SMTP host、port、TLS／SSL、帳密、寄件人政策、防火牆與 relay 是否允許該主機。若 Email 通而 RTSP 不通，檢查攝影機帳密、VLAN、RTSP path、codec 與 OpenCV／FFmpeg 支援。

11. 啟動現場監測

先以前景方式跑，觀察 log：

python scripts/run_monitor.py --settings configs/monitor_settings.env

主要輸出：

data/runtime/logs/monitor.log
data/runtime/detections.jsonl
data/runtime/evidence/YYYY-MM-DD/*.jpg
data/runtime/alerts/YYYY-MM-DD/*_smoke.jpg
data/runtime/alerts/YYYY-MM-DD/*_smoke.json
http://127.0.0.1:8765/             # ALLOW_LIVE_STREAMING=true 時

JSONL 紀錄包含時間、分類、conf、area、ROI、box、mask polygon、alarm state、事件與 Email 成功／失敗狀態。為避免無煙時每一張影格都造成檔案膨脹，預設只保存 smoke 偵測與告警／解除事件；若需要稽核無煙狀態，可將 NO_SMOKE_LOG_INTERVAL_SECONDS 設為大於 0 的秒數，按間隔保存一筆 no_smoke_summary。告警 JSON sidecar 也保留相同資訊，供 報警資料檢查／重標註 GUI 使用。舊設定名 SAVE_POSITIVE_SNAPSHOTS 仍可相容讀取，但新設定請使用 SAVE_ALERT_SNAPSHOTS。

Shadow mode：將 SHADOW_MODE=true 後，程式會照常保存煙霧與告警事件、更新狀態機並在 JSONL 標記 suppressed_shadow_mode，但不會呼叫 SMTP。確認現場結果後再改回 false。

既有檔案：新設定只影響之後的寫入，不會自動縮小既有的 data/runtime/detections.jsonl。若舊檔案已過大，請先停止 monitor，再將它移到已備份的日期資料夾或改名封存，程序會建立新的 JSONL 檔。

Live video：設定 ALLOW_LIVE_STREAMING=true 後，用 Chrome 或 Edge 開啟 http://127.0.0.1:8765/；頁面會以 HTML5 <video> 元件顯示影片，不使用 <img> 播放。若使用 LIVE_STREAM_HOST=0.0.0.0，請以現場電腦實際 IP 取代網址中的 127.0.0.1，並限制防火牆來源。影格只保留在記憶體，不會產生逐影格 JPG 檔案。

12. 用 GUI 複核告警與回送資料集

告警影像與訓練資料集是兩個不同位置。安裝輕量依賴並啟動 GUI：

python -m pip install -r alert_reviewer/requirements.txt
python alert_reviewer/gui/alert_reviewer_gui.py \
  --input data/runtime/alerts \
  --dataset data/processed/field_yolo \
  --split train

在影像上檢查原模型的 ROI、box、mask、conf、area、cls，必要時重畫 polygon，再選 smoke 或 no_smoke。完成後可以保留在原地、移動到 YOLO train/val/test，或刪除。GUI 不會載入模型，也不會寄 Email。

移動成功後才會移除該筆告警 JPG／JSON；刪除前請確認已有備份。把資料送回訓練前，按攝影機／日期／影片重新規劃 split，避免連續影格洩漏。

完整操作請閱讀 alert_reviewer 操作說明；子專案架構見 alert_reviewer 架構。

14. 門檻調整方法

一次只改一個因素，使用同一批已標記影片比較。建議順序：

  1. 先固定 ROI，排除天空與非煙囪區域；
  2. 用 CONF_THRESHOLD 調整模型嚴格度；
  3. 用 MIN_SMOKE_AREA_RATIO 排除太小的雜訊；
  4. 用 POSITIVE_FRAMES_TO_ALARM 抑制短暫誤判；
  5. 用 NEGATIVE_FRAMES_TO_CLEAR 避免煙霧短暫消失就解除；
  6. 仍漏檢時，加入現場資料重新標註與微調，或比較 YOLO11s-seg。
現象優先處理
雲、蒸汽或樹影常誤報縮小／修正 ROI、提高 confidence 或 area、增加連續 positive 門檻、補充 hard negative 標註資料。
很淡的煙霧漏檢確認標註是否包含淡煙、降低 confidence／area、提高影像尺寸，並以現場資料微調。
告警開始太慢降低 positive frames 或縮短抽樣間隔，但要重新估計誤報率。
告警解除太快增加 negative frames；不要把模型每一次短暫 negative 都直接當作無煙。
每秒寄信確認 ALERT_REPEAT_SECONDS 與程式是否正常進入 ALARM；預設只在開始與冷卻後提醒。

15. 維運、備份與版本回復

  • 每天或每週備份 data/runtime/detections.jsonl、告警 evidence 與 configs/monitor_settings.env（加密）；
  • 每次換模型使用版本名，例如 smoke_yolo11n_20260717_v001.pt，並在變更紀錄記下資料集、權重、門檻與驗證結果；
  • 新模型先在 replay video 與現場 shadow mode 驗證，再取代 models/best.pt；
  • 保留上一版可運作模型，以便把 MODEL_PATH 改回去；
  • 監看 RTSP 斷線、模型 inference error、SMTP failed 與磁碟剩餘容量。

本專案會保留候選影像，但不會自動把未經人工確認的資料拿來重訓。自動自我訓練可能把誤判放大，應由人員標註後再建立新版本。

16. 常見問題

為什麼完全隔離外網仍然要求 SMTP？

Email 本身是一種網路服務。可以不連外網，但必須存在一條到內部 SMTP relay 的路徑；若連內部郵件伺服器也沒有，程式無法把訊息送出，只能先寫入本地 outbox 或由其他系統轉送。

OpenCV 顯示無法開啟 RTSP，應先改模型嗎？

不用。先用 check_rtsp.py、檢查 IP／帳密／path、VLAN、防火牆與 codec。這是串流連線問題，不是 YOLO 訓練問題。程式預設要求 RTSP over TCP，因為在工業網路通常比 UDP 穩定。

IJmond 的 camera split 分數很高，代表現場一定準嗎？

不代表。camera split 只說明在 IJmond 的攝影機分布下泛化能力較好；你的煙囪、鏡頭與天氣仍是 domain shift。現場影片驗收與本地微調是必要步驟。

要不要把低煙與高煙分成兩個類別？

若目前只需「有沒有冒煙」，合併成一類更簡單。若之後要做不透明度分級，應重新設計標註規則、類別不平衡處理與法規驗證，不要直接把第一版的 confidence 當成不透明度百分比。

怎麼知道告警真的寄出去了？

使用 test_email.py 做部署前測試，並查看 detections.jsonl 中的 email_events 與 monitor.log。實務上仍應由收件端確認郵件是否進入信箱或被 relay 拒收。
