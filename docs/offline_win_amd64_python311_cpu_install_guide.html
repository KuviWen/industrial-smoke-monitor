隔離 Windows 電腦
Python 3.11.15 與 YOLO11 CPU 環境離線安裝操作手順

在可連網的 Linux 電腦上準備 Windows AMD64 專用 Conda 套件與 Python wheel，再於隔離外網的 Windows 電腦透過 Anaconda Prompt 建立環境並完成離線安裝。

準備端：Linux 目標端：Windows win-amd64 Python 3.11.15 PyTorch 2.6.0 CPU Ultralytics YOLO11
本文件涵蓋現場執行：這是「現場仍要安裝 Python／Conda」的完整離線安裝手順。即使現場只做推論、不重新訓練，仍需依本文件建立 Python 3.11.15 環境，並在完成後執行模型、RTSP 與 Email 驗收。
先理解模型：需要先認識 YOLO11 的模型任務、預訓練、評估指標與部署格式時，請參考 YOLO11 特色與工作原理導覽。

1. 目的、範圍與重要原則

本手順適用於下列固定條件：

項目固定條件
可連網準備電腦Linux；需有可用的 Conda，以及任一可執行 python3 -m pip 的既有 Python。無須另建 wheel 下載環境。
隔離目標電腦Windows 64 位元，Conda 平台應顯示 win-64，pip 平台為 win_amd64。
Python3.11.15
PyTorchtorch==2.6.0+cpu、torchvision==0.21.0+cpu；不使用 CUDA。
正式相依檔requirements.txt
開發／驗收相依檔requirements-dev.txt；內含 -r requirements.txt 並額外安裝 pytest。
禁止直接搬移 Linux 環境。 Linux 上建立的 Conda 環境、Python 執行檔或 conda-pack 壓縮檔不能交給 Windows 使用。Linux 端只能利用目標平台參數，下載 Windows 專用的 Conda 套件與 wheel。
不可只下載最上層套件。 Ultralytics、OpenCV、PyTorch 等均有間接相依套件；必須先針對 Windows 目標完整解析並鎖定，再下載鎖定清單內的全部 wheel，最後將整個 conda_pkgs 和 wheelhouse 帶入隔離電腦。
命名要求。 兩份檔案在整個流程中固定命名為 requirements.txt 與 requirements-dev.txt，並放在同一資料夾。不要改成 requirements-win.txt 或其他名稱。

2. 確認 requirements

2.1 requirements.txt

母專案根目錄的 requirements.txt（GPU 預設版）應為以下內容：

# Install a CUDA-compatible torch/torchvision build first on a GPU training PC.
# The exact torch wheel must match the installed NVIDIA driver and the chosen
# CUDA runtime. The field PC can use the same environment or a CPU-only build.
ultralytics>=8.3.0,<9.0
opencv-python>=4.9.0,!=4.13.0.90
numpy==1.26
PyYAML>=6.0
python-dotenv>=1.0
Pillow>=10.0

# Annotate the following lines if no GPUs.
torch==2.6.0+cu124
torchvision==0.21.0+cu124

# Un-annotate the following lines if no GPUs.
# torch==2.6.0+cpu
# torchvision==0.21.0+cpu

本手順的目標端是沒有 GPU 的 Windows CPU 電腦。下方 3.1 把檔案複製到 bundle 後、執行 uv 解析或 pip 下載前，會先把 CUDA 行註解並取消 CPU 行的註解；執行切換命令時請先位於該 bundle 目錄，不要直接修改 Git repository 的根目錄檔案。

sed -i \
  -e 's/^torch==2.6.0+cu124$/# torch==2.6.0+cu124/' \
  -e 's/^torchvision==0.21.0+cu124$/# torchvision==0.21.0+cu124/' \
  -e 's/^# torch==2.6.0+cpu$/torch==2.6.0+cpu/' \
  -e 's/^# torchvision==0.21.0+cpu$/torchvision==0.21.0+cpu/' \
  "requirements.txt"

這個切換只改離線 bundle 內的副本，不會改 Git repository 的根目錄檔案。切換後有效的 Torch 版本是 torch==2.6.0+cpu 與 torchvision==0.21.0+cpu；numpy==1.26 與標準 opencv-python 則維持不變。

2.2 requirements-dev.txt

-r requirements.txt
pytest>=8.0
檔案狀態正確。 新版 requirements 使用標準 opencv-python、固定 numpy==1.26，並把 GPU／CPU 的 Torch 選項放在同一份 requirements.txt。本流程不需要另外建立 Torch requirements 檔。
requirements 開頭三行仍在說明 GPU 訓練電腦的選版原則，但註解不會參與 pip 解析；完成上方切換後，真正生效的是檔案末尾的 torch==2.6.0+cpu 與 torchvision==0.21.0+cpu，因此本手順建立的是純 CPU 環境。

若瀏覽器下載後自動加上 (2) 等後綴，進入操作流程前必須重新命名回上述兩個標準名稱。

階段 A3. Linux 連網電腦：建立離線包

3.1 建立工作目錄並放入 requirements

以下命令使用 Bash。先建立固定目錄：

WORK="$HOME/offline_build"
BUNDLE="$WORK/offline-smoke-win64-py31115-cpu"

mkdir -p "$BUNDLE/conda_pkgs"
mkdir -p "$BUNDLE/wheelhouse"

把附件複製到 bundle，並確保名稱正確。請將來源路徑替換成實際位置：

cp "/path/to/requirements.txt" "$BUNDLE/requirements.txt"
cp "/path/to/requirements-dev.txt" "$BUNDLE/requirements-dev.txt"

如果下載後檔名含有 (2)，可直接在複製時改回標準名稱：

cp "/path/to/requirements(2).txt" "$BUNDLE/requirements.txt"
cp "/path/to/requirements-dev(2).txt" "$BUNDLE/requirements-dev.txt"

現在進入 bundle 目錄，執行 2.1 顯示的 CPU Torch 切換命令；該命令的最後一個檔名是 requirements.txt，所以只會修改 bundle 副本：

cd "$BUNDLE"

確認兩個檔案存在且位於同一資料夾：

ls -l "$BUNDLE/requirements.txt" "$BUNDLE/requirements-dev.txt"
sed -n '1,120p' "$BUNDLE/requirements.txt"
sed -n '1,80p' "$BUNDLE/requirements-dev.txt"

3.2 確認 Linux 端 Conda

conda --version
conda info

這裡顯示 Linux 平台是正常的；後續命令會另外指定目標平台 win-64。

3.3 下載 Windows AMD64 的 Python 3.11.15 Conda 套件

指定獨立套件快取，確保帶走的是本次完整下載結果：

export CONDA_PKGS_DIRS="$BUNDLE/conda_pkgs"
若正在重做 3.3，先使用全新的 conda_pkgs。 不要把不同求解結果累積在同一資料夾。可先把舊資料夾重新命名保留，再建立空白資料夾： 
mv "$BUNDLE/conda_pkgs" \
  "$BUNDLE/conda_pkgs.previous-$(date +%Y%m%d-%H%M%S)"
mkdir -p "$BUNDLE/conda_pkgs"
export CONDA_PKGS_DIRS="$BUNDLE/conda_pkgs"
第一次執行且資料夾原本為空時，不需執行這一段。

跨平台解析並下載 Windows 套件；--download-only 會在下載完成後停止，不會嘗試於 Linux 執行 Windows Python：

conda create \
  -n _download_win31115 \
  --platform win-64 \
  --download-only \
  --override-channels \
  -c defaults \
  --no-default-packages \
  python=3.11.15 \
  pip \
  setuptools \
  wheel \
  -y

下載完成後取消臨時環境變數：

unset CONDA_PKGS_DIRS
若 Conda 回報不認得 --platform，把該參數替換成 --subdir win-64 後重試；其餘參數不變。

確認 Python 3.11.15 套件和其他 Conda 套件封裝檔已下載：

find "$BUNDLE/conda_pkgs" -maxdepth 1 \
  -name 'python-3.11.15-*' -print

find "$BUNDLE/conda_pkgs" -maxdepth 1 -type f \
  \( -name '*.conda' -o -name '*.tar.bz2' \) | wc -l

第一段必須找到 Python 3.11.15；第二段數量必須大於 0。

3.4 準備 Linux 建包工具（不建立 Conda 環境）

uv pip compile 只解析並輸出鎖定檔，不會安裝套件到目前 Python，因此不需要另外建立 wheel_downloader Conda 環境。把新版 pip 與 uv 安裝到 bundle 外的私有工具目錄即可：

TOOLS="$WORK/linux_build_tools"
MODULES="$TOOLS/python_modules"
UV_CACHE="$WORK/linux_uv_cache"
mkdir -p "$MODULES" "$UV_CACHE"

python3 --version
python3 -m pip --version

python3 -m pip install \
  --upgrade \
  --no-cache-dir \
  --target "$MODULES" \
  pip \
  uv

PYTHONPATH="$MODULES" \
python3 -m pip --version

UV_CACHE_DIR="$UV_CACHE" \
PYTHONPATH="$MODULES" \
python3 -m uv --version
不修改 Conda base，也不建立新環境。 --target "$MODULES" 只把新版 pip 與 Linux 版 uv 放入本次工作目錄；TOOLS 位於 bundle 外，因此不會被打包帶入 Windows。後續下載也使用這份私有 pip，不受 Linux 系統 pip 版本影響。
不要直接輸入 uv。 後續固定透過 PYTHONPATH="$MODULES" 執行私有 pip/uv，因此不依賴 Bash 的 PATH。uv 快取也固定放在可寫入的 UV_CACHE。若主機只有 python 而沒有 python3，本節及後續 Linux 命令中的 python3 全部一致替換成 python。

3.5 先鎖定 Windows 相依，再下載全部 wheel

直接用 pip 跨平台解析會誤判 Torch 條件。 pip download --platform win_amd64 會依 Windows 篩選 wheel，但套件的環境條件仍可能依執行 pip 的 Linux 主機判斷。因此 Torch 的 platform_system == "Linux" 條件被視為成立，接著 pip 又只准下載 Windows wheel，最後就會錯誤要求 nvidia-nccl-cu12。這不表示 CPU 版 Torch 需要 CUDA，也不要手動加入 NVIDIA CUDA 套件。

3.5.1 若舊版 Step 3.5 已失敗：先保留並隔離殘留檔案

只有已執行過舊命令且失敗時才做這一步；新流程首次執行可跳過：

mv "$BUNDLE/wheelhouse" \
  "$BUNDLE/wheelhouse.failed-$(date +%Y%m%d-%H%M%S)"
mkdir -p "$BUNDLE/wheelhouse"

這是可回復的重新命名，不會刪除原本下載內容。

3.5.2 用 uv 針對 Windows AMD64 / Python 3.11.15 解析完整鎖定清單

cd "$BUNDLE"

UV_CACHE_DIR="$UV_CACHE" \
PYTHONPATH="$MODULES" \
python3 -m uv pip compile \
  --no-python-downloads \
  --python-version 3.11.15 \
  --python-platform x86_64-pc-windows-msvc \
  --default-index https://pypi.org/simple \
  --find-links https://download.pytorch.org/whl/cpu/torch/ \
  --find-links https://download.pytorch.org/whl/cpu/torchvision/ \
  --only-binary :all: \
  --output-file "$BUNDLE/requirements-win64-py311.lock" \
  "$BUNDLE/requirements-dev.txt"

x86_64-pc-windows-msvc 明確代表 64 位元 Windows。uv 會使用 Windows 的環境條件完成解析，並把正式、開發及間接相依套件全部輸出成精確版本。PyPI 是唯一的一般套件索引；兩個 --find-links 只補充官方的 Torch 與 Torchvision CPU wheel 清單。

若 Linux 現有 Python 不是 3.11.15，uv 可能提示使用現有版本作為建包工具。這不會改變目標條件：解析仍明確指定 Python 3.11.15 與 Windows AMD64，且 --only-binary :all: 禁止來源碼建置。

先確認鎖定檔包含 CPU 版 Torch，而且沒有誤列 CUDA 執行期套件：

grep -E '^(torch|torchvision)==' \
  "$BUNDLE/requirements-win64-py311.lock"

grep -E '^(nvidia-(nccl|cusparselt)-cu12|triton)==' \
  "$BUNDLE/requirements-win64-py311.lock"

第一段應顯示 torch==2.6.0+cpu 與 torchvision==0.21.0+cpu；第二段應無輸出。鎖定檔出現 nvidia-ml-py 是 Ultralytics 的跨平台監控相依，並非 CUDA runtime，可保留。

3.5.3 按鎖定清單下載 Windows wheel

PYTHONPATH="$MODULES" python3 -m pip download \
  --dest "$BUNDLE/wheelhouse" \
  --platform win_amd64 \
  --python-version 311 \
  --implementation cp \
  --abi cp311 \
  --abi abi3 \
  --abi none \
  --only-binary=:all: \
  --no-deps \
  --index-url https://pypi.org/simple \
  --find-links https://download.pytorch.org/whl/cpu/torch/ \
  --find-links https://download.pytorch.org/whl/cpu/torchvision/ \
  -r "$BUNDLE/requirements-win64-py311.lock"

requirements-dev.txt 會在前一步遞迴讀取同目錄的 requirements.txt。uv 產生的鎖定檔已逐項列出完整相依，因此這一條命令會下載：

  • 正式環境所需的全部套件；
  • torch==2.6.0+cpu 與 torchvision==0.21.0+cpu；
  • Ultralytics 的間接相依套件；
  • pytest 及其相依套件。
不要移除 --no-deps 或 --only-binary=:all:。 --no-deps 在這裡是安全且必要的，因為完整相依已由 uv 寫入鎖定檔；它可防止 pip 再次依 Linux 主機解析 Torch metadata。若 --only-binary=:all: 導致失敗，代表某個套件缺少目標 Windows wheel，不能在 Linux 上下載原始碼後期待 Windows 離線電腦自行編譯。

3.6 檢查 wheelhouse

確認沒有誤抓 Linux 或 macOS wheel：

find "$BUNDLE/wheelhouse" -maxdepth 1 -type f \
  \( -name '*manylinux*' -o -name '*linux_*' -o -name '*macosx*' \) \
  -print

正常結果應為空白。

確認沒有來源封裝檔：

find "$BUNDLE/wheelhouse" -maxdepth 1 -type f \
  ! -name '*.whl' -print

正常結果亦應為空白。

確認關鍵 Windows wheel：

find "$BUNDLE/wheelhouse" -maxdepth 1 \
  \( -iname 'torch*win_amd64.whl' \
     -o -iname 'torchvision*win_amd64.whl' \
     -o -iname 'opencv*win_amd64.whl' \
     -o -iname 'numpy*win_amd64.whl' \) \
  -print | sort

其中應包含 CPU 版 torch/torchvision，檔名通常類似：

torch-2.6.0+cpu-cp311-cp311-win_amd64.whl torchvision-0.21.0+cpu-cp311-cp311-win_amd64.whl

3.7 在 Linux 上進行離線 wheel 齊全性預檢

相依關係已由 uv 針對 Windows 解析；這一步按鎖定檔逐項驗證本地 wheel 是否齊全，不會執行 Windows DLL：

PYTHONPATH="$MODULES" python3 -m pip install \
  --dry-run \
  --ignore-installed \
  --no-deps \
  --no-index \
  --find-links="$BUNDLE/wheelhouse" \
  --platform win_amd64 \
  --python-version 311 \
  --implementation cp \
  --abi cp311 \
  --abi abi3 \
  --abi none \
  --only-binary=:all: \
  -r "$BUNDLE/requirements-win64-py311.lock"

3.8 最終離線包結構

offline-smoke-win64-py31115-cpu/ ├── conda_pkgs/ │ ├── cache/ │ ├── python-3.11.15-... │ ├── pip-... │ └── 其他 Conda 套件與封裝檔 ├── wheelhouse/ │ ├── torch-2.6.0+cpu-...-win_amd64.whl │ ├── torchvision-0.21.0+cpu-...-win_amd64.whl │ ├── ultralytics-... │ ├── opencv_python-... │ └── 所有其他 wheel ├── requirements-win64-py311.lock ├── requirements.txt └── requirements-dev.txt

階段 B4. 跨網域傳輸與完整性檢查

4.1 建立 Windows 可直接解壓的 ZIP

使用 Python 標準函式庫建立 ZIP，不依賴 Linux 的 zip 命令，也不要求 Windows 提供 tar：

cd "$WORK"
python3 -m zipfile -c \
  offline-smoke-win64-py31115-cpu.zip \
  offline-smoke-win64-py31115-cpu

python3 -m zipfile -t \
  offline-smoke-win64-py31115-cpu.zip

測試命令不可回報損壞或 CRC 錯誤。Python 的 ZIP64 支援可處理超過 4 GiB 的 ZIP，因此大型 Torch wheelhouse 亦可使用。

4.2 產生 SHA-256

sha256sum offline-smoke-win64-py31115-cpu.zip \
  > offline-smoke-win64-py31115-cpu.zip.sha256

透過公司核准的隨身碟或媒體帶入以下兩個檔案：

  • offline-smoke-win64-py31115-cpu.zip
  • offline-smoke-win64-py31115-cpu.zip.sha256
離線包不得包含 RTSP 密碼、SMTP 密碼、API 金鑰或正式環境的 configs/monitor_settings.env。機敏設定應依現場資訊安全流程另外配置。

階段 C5. Windows 隔離電腦：離線安裝

以下所有命令均在 Anaconda Prompt 中執行。

5.1 確認 Windows 與 Conda 平台

where conda
conda --version
conda info | findstr /I "platform"

平台必須顯示：

platform : win-64

5.2 驗證傳輸檔 SHA-256

假設外接媒體是 E:：

certutil -hashfile E:\offline-smoke-win64-py31115-cpu.zip SHA256
type E:\offline-smoke-win64-py31115-cpu.zip.sha256

人工比較兩者的 64 位十六進位雜湊值，必須完全相同。不同時不得繼續安裝，應重新傳輸。

5.3 使用 Anaconda 的 Python 解壓至純英文路徑

以下使用 C:\Users\Public\offline-install；如公司權限政策不允許，請換成其他可寫入且不含中文的本機路徑。先測試 ZIP，再以 Python 標準函式庫解壓：

where python
python -m zipfile -t E:\offline-smoke-win64-py31115-cpu.zip

mkdir C:\Users\Public\offline-install
python -m zipfile -e ^
  E:\offline-smoke-win64-py31115-cpu.zip ^
  C:\Users\Public\offline-install
不需要 tar 或 7-Zip。 此處使用 Anaconda Prompt 原本就能執行的 base Python；zipfile 是 Python 3.11 標準函式庫，不需額外安裝套件或連線。
若舊版 .tar.gz 已經帶入隔離電腦，不必重新傳輸。 先完成 5.2 的 SHA-256 比對，再以 Anaconda 的 Python 標準函式庫測試及解壓： 
where python
python -m tarfile -t E:\offline-smoke-win64-py31115-cpu.tar.gz

mkdir C:\Users\Public\offline-install
python -m tarfile -e ^
  E:\offline-smoke-win64-py31115-cpu.tar.gz ^
  C:\Users\Public\offline-install
此段只供既有舊包應急；下次建包仍應依 4.1 使用 ZIP。

設定本次工作變數：

set "BUNDLE=C:\Users\Public\offline-install\offline-smoke-win64-py31115-cpu"
set "SOURCE_PKGS=%BUNDLE%\conda_pkgs"

確認必要檔案：

dir "%BUNDLE%\requirements.txt"
dir "%BUNDLE%\requirements-dev.txt"
dir "%BUNDLE%\wheelhouse\*.whl"
dir "%BUNDLE%\conda_pkgs\python-3.11.15*"

5.4 建立 Python 3.11.15 Conda 環境

不要再執行舊版命令。 若命令中仍有 --use-index-cache、python=3.11.15、pip、setuptools、wheel，表示仍在使用舊版 Step 5.4。該命令可能直接使用從 Linux 搬來、但在傳輸後缺檔的已解壓資料夾，因而在 Verifying transaction 發生 CondaVerificationError。以下新版流程只以原始 .conda／.tar.bz2 封裝檔建立環境。
python=3.11.15 並不是要求它預先存在。 一般 Conda 求解中，這代表要安裝進新環境的目標版本。不過，跨作業系統搬移 package cache 後，快取 repodata 可能無法被 Windows 求解器可靠識別；本手順改由所有已下載的本地封裝檔建立 @EXPLICIT 清單，不再依賴離線 repodata 求解。

5.4.1 確認關鍵 Conda 封裝檔確實存在

dir /b "%SOURCE_PKGS%\python-3.11.15-*"
dir /b "%SOURCE_PKGS%\pip-*"
dir /b "%SOURCE_PKGS%\setuptools-*"
dir /b "%SOURCE_PKGS%\wheel-*"

四項都必須至少顯示一個 .conda 或 .tar.bz2 封裝檔。若任何一項顯示 File Not Found，代表 Linux 階段 3.3 的下載包不完整，不能靠刪除版本條件解決，必須重建並重新傳輸 Conda 套件包。

5.4.2 從本地封裝檔生成 Windows 專用 explicit 清單

來源封裝與 Conda 的執行快取分開，並以隨機尾碼強制建立全新的執行快取。這一步可避開先前報錯的已解壓套件資料夾：

set "CONDA_PKGS_DIRS=%USERPROFILE%\conda_pkgs_smoke311_%RANDOM%%RANDOM%"
mkdir "%CONDA_PKGS_DIRS%"
echo %CONDA_PKGS_DIRS%

python -c "from pathlib import Path; p=Path(r'%SOURCE_PKGS%'); f=sorted(list(p.glob('*.conda'))+list(p.glob('*.tar.bz2'))); required=('python-3.11.15-','pip-','setuptools-','wheel-'); missing=[n for n in required if not any(x.name.startswith(n) for x in f)]; assert f, 'No .conda or .tar.bz2 package archives found'; assert not missing, 'Missing package archives: '+', '.join(missing); o=Path(r'%BUNDLE%\conda-win64-explicit-local.txt'); o.write_text('@EXPLICIT\n'+'\n'.join(x.resolve().as_uri() for x in f)+'\n', encoding='utf-8'); print('package archives =', len(f)); print('explicit file =', o)"
不要把 CONDA_PKGS_DIRS 設回 %SOURCE_PKGS%。 SOURCE_PKGS 是唯讀來源；新的 CONDA_PKGS_DIRS 是讓 Conda 從原始封裝重新解壓的工作快取。若畫面中的 package location 仍指向 %SOURCE_PKGS%\setuptools-..._0 這種無副檔名資料夾，請中止，關閉 Anaconda Prompt 後依 5.3～5.4 重新設定變數。

套件數必須大於 0，並確認清單包含四個關鍵套件：

findstr /I "python-3.11.15-" "%BUNDLE%\conda-win64-explicit-local.txt"
findstr /I "pip-" "%BUNDLE%\conda-win64-explicit-local.txt"
findstr /I "setuptools-" "%BUNDLE%\conda-win64-explicit-local.txt"
findstr /I "wheel-" "%BUNDLE%\conda-win64-explicit-local.txt"

5.4.3 直接從 explicit 清單建立環境

conda create ^
  -n smoke311_cpu ^
  --offline ^
  --file "%BUNDLE%\conda-win64-explicit-local.txt" ^
  -y
此命令不再另外寫 python=3.11.15 pip setuptools wheel。 原因不是這些規格錯誤，而是 explicit 清單已經逐一指定包含它們在內的所有確切封裝檔；Conda 的 explicit 模式不應再混用套件名稱規格。

啟用並確認 Python：

conda activate smoke311_cpu
python --version
where python
python -c "import sys, platform; print(sys.executable); print(platform.machine()); print(sys.maxsize > 2**32)"

預期結果：

  • Python 3.11.15
  • platform.machine() 顯示 AMD64 或等價 x86_64 表示
  • 最後一行為 True，表示 64 位元 Python

5.5 正式部署環境安裝

正式現場環境使用 requirements.txt：

python -m pip install ^
  --no-index ^
  --find-links="%BUNDLE%\wheelhouse" ^
  --only-binary=:all: ^
  -r "%BUNDLE%\requirements.txt"

5.6 含 pytest 的開發／驗收環境安裝

如果此環境需要執行 pytest，請改用以下命令：

python -m pip install ^
  --no-index ^
  --find-links="%BUNDLE%\wheelhouse" ^
  --only-binary=:all: ^
  -r "%BUNDLE%\requirements-dev.txt"
二擇一即可。 requirements-dev.txt 已包含 -r requirements.txt，因此執行開發／驗收命令時，不必先執行正式安裝命令。Linux 端 wheelhouse 已預先包含兩者所需的全部 wheel。
安裝命令使用 --no-index，可保證 pip 不會嘗試連接 PyPI 或其他網路索引，只會使用本地 wheelhouse。

6.6 使用本地模型做 YOLO CPU 推論

requirements 只包含程式套件，不包含 YOLO 權重。必須另外把訓練完成的 best.pt 和測試影像帶入現場。

假設：

  • 模型：C:\industrial-smoke-monitor\models\best.pt
  • 測試影像：C:\industrial-smoke-monitor\tests\smoke_test.jpg
python -c "from ultralytics import YOLO; model=YOLO(r'C:\industrial-smoke-monitor\models\best.pt'); results=model.predict(source=r'C:\industrial-smoke-monitor\tests\smoke_test.jpg', device='cpu', save=False, verbose=False); print('result_count =', len(results)); print('YOLO CPU inference OK')"
模型路徑必須指向已存在的本地檔案。若只寫 YOLO('yolo11n.pt') 且本機沒有該檔，Ultralytics 可能嘗試連網下載，隔離環境將失敗。

8. 常見錯誤排除

錯誤／現象可能原因處理方式
'tar' is not recognized／tar 不是內部或外部命令 隔離 Windows 未提供 tar.exe，或它不在 PATH。 不要安裝額外解壓工具。既有 .tar.gz 可用 5.3 的 python -m tarfile 應急；下次依 4.1 改建 ZIP，並用 python -m zipfile 解壓。
bash: uv: command not found 直接呼叫了不在 PATH 的 uv 執行檔。 不必建立或啟用任何 Conda 工具環境。先重做 3.4，再透過 PYTHONPATH="$MODULES" python3 -m uv ... 執行，不要使用裸命令 uv ...。
No module named uv MODULES 未設定、私有工具目錄尚未安裝 uv，或 PYTHONPATH 指向錯誤。 重設 3.4 的 WORK、TOOLS、MODULES，重新把 pip uv 安裝到 --target "$MODULES"，再以 3.4 的版本命令確認。
Failed to initialize cache 或 uv cache Permission denied 預設的 ~/.cache/uv 不可寫入。 依 3.4 設定 UV_CACHE="$HOME/offline_build/linux_uv_cache"，並在 uv 命令前保留 UV_CACHE_DIR="$UV_CACHE"。
PackagesNotFoundError 或 Conda 找不到 Python 3.11.15 跨作業系統搬移後，Windows 求解器未能從快取 repodata 辨識套件；或離線包實際缺少封裝檔。即使畫面列出 defaults 的網址，也不代表 --offline 已連網。 不要刪除 python=3.11.15 來規避。先依 5.4.1 檢查四種封裝檔；存在時依 5.4.2～5.4.3 建立本地 @EXPLICIT 清單並安裝。若任何封裝檔缺失，才回 Linux 以全新的 conda_pkgs 重做 3.3 並重新傳輸。
CondaVerificationError，並指出 setuptools\_vendor\.lock 或其他 manifest 路徑不存在 Conda 正在使用搬入後缺檔的「已解壓 package cache 資料夾」。這不是 Python 版本或相依求解錯誤；本例已在 Solving environment 完成後才失敗。 不要關閉驗證，也不要再次執行含 --use-index-cache 的舊命令。依 5.4.1 確認原始 .conda／.tar.bz2 封裝存在，再依 5.4.2 使用新的隨機 CONDA_PKGS_DIRS，最後以 5.4.3 的 --file 命令重新解壓並建立環境。若仍在全新快取報相同缺檔，原始封裝或傳輸檔已損壞，須重做 3.3 並重新比對 ZIP SHA-256。
No matching distribution found for torch==2.6.0+cpu Linux 下載時缺少 PyTorch CPU wheel 來源，或 wheelhouse 沒有 Windows AMD64 CPU wheel。 確認 uv 與 pip 下載命令均含官方 Torch、Torchvision CPU 的兩個 --find-links，並檢查 torch-2.6.0+cpu-...-win_amd64.whl。
No matching distribution found for nvidia-nccl-cu12，並顯示 platform_system == "Linux" 在 Linux 上直接用 pip 解析 Windows Torch，相依 marker 仍依 Linux 主機判斷，與 --platform win_amd64 形成衝突。 不要安裝 NCCL、不要改成 CUDA Torch。依 3.5 先用 uv 的 --python-platform x86_64-pc-windows-msvc 產生完整鎖定檔，再以 pip --no-deps 下載鎖定清單。
is not a supported wheel on this platform 誤帶 Linux wheel、32 位元 Python，或 Python 版本不是 3.11。 確認 Conda 顯示 win-64、Python 是 64 位元 3.11.15，並重新以 --platform win_amd64 --python-version 311 下載。
pip 安裝時嘗試連網 漏掉 --no-index，或執行了普通 pip install -r ...。 取消命令，改用本手順的 --no-index --find-links="...\wheelhouse"。
Could not open requirements file: requirements.txt requirements-dev.txt 與 requirements.txt 不在同一資料夾，或名稱被加上 (2)。 確認兩者在 bundle 根目錄，且名稱完全等於 requirements.txt 與 requirements-dev.txt。
ImportError、DLL load failed wheel 損壞、平台錯誤、Microsoft Visual C++ Runtime 缺失，或存在多套 OpenCV。 先比對 SHA-256；確認只安裝 opencv-python；依公司規範確認 Microsoft Visual C++ 2015–2022 x64 Redistributable 已安裝。
UnicodeDecodeError: cp950 繁體中文 Windows 編碼、中文路徑或套件腳本輸出編碼不一致。 使用純英文路徑；在同一 Anaconda Prompt 先執行 set PYTHONUTF8=1 及 set PYTHONIOENCODING=utf-8 後重試。
torch.cuda.is_available() == False 本方案安裝 CPU build。 這是預期結果，不需處理。若未來改為 GPU 部署，必須重新建立獨立 GPU wheelhouse 與環境。

  • Linux 建包流程不依賴或啟用 wheel_downloader Conda 環境；私有 pip/uv 工具位於 bundle 外。
