報警資料檢查與重標註：子專案架構

這個工具處理的是「模型曾經報警的候選資料」，不是原始訓練資料，也不是直接修改模型。它讓人員先看證據，再決定是否把修正後的樣本送回訓練流程。

設計原則：告警資料先寫到 data/runtime/alerts/，與 data/processed/ 分開；只有使用者按下「移動到訓練資料集」才會進入 YOLO 資料結構。

1. 資料流

monitor
影像＋評分
→
alerts/
JPG＋JSON sidecar
→
Tkinter GUI
查看／重畫 polygon
→
人工決策
保留／移動／刪除

3. 告警 artifact

檔案用途保留內容
YYYY-MM-DD/*_smoke.jpg監控當下的標註影像原圖、ROI 框、box、mask、conf、area、cls、instances
YYYY-MM-DD/*_smoke.json可讀的評分與 provenance時間、攝影機、模型判斷、polygon、alarm state、shadow mode、檔案路徑
同名 .txt人工複核後的 YOLO segmentation labelsmoke 為 polygon；no_smoke 為空檔

JSON 的 instances 是模型原始結果；GUI 的 review.polygons 才是人工修正後的結果。兩者同時保存，方便事後比較模型與人工。

4. 與母專案的邊界

元件負責不負責
母專案 monitorRTSP、YOLO11-seg、時間狀態機、Email、保存 alert自動把未確認影像拿去重訓
alert reviewer查看、重畫 polygon、建立 parent-compatible YOLO sample推論、訓練或寄信
dataset_builder影片抽幀、標註與資料集管理決定告警是否為真煙霧

5. 輸出結構

data/processed/field_yolo/
├── dataset.yaml
├── manifest.csv
├── images/train/*.jpg
├── labels/train/*.txt
├── images/val/*.jpg
└── labels/val/*.txt

此結構可直接交給母專案的 configs/ijmond.yaml 同類 YOLO segmentation 流程使用；實際訓練前仍要按攝影機、日期或場景重新規劃 train/val/test，避免相鄰影格洩漏到不同 split。
